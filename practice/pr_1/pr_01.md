## Создание первой базы данных и таблицы

### Создание базы данных (В облачном сервисе БД предоставлена по умолчанию)

```sql
CREATE DATABASE IF NOT EXISTS my_first_db;

USE my_first_db;
```

### Создание таблицы для логов

```sql
CREATE TABLE access_logs (
    timestamp    DateTime64(3),
    event_type   LowCardinality(String),
    user_id      UInt64,
    ip_address   IPv4,
    url          String,
    duration_ms  UInt32
)
ENGINE = MergeTree()
ORDER BY (timestamp, user_id);
```

**Разбор столбцов и типов данных:**

| Столбец | Тип | Пояснение |
|---------|-----|-----------|
| `timestamp` | `DateTime64(3)` | Дата и время с точностью до миллисекунд (3 знака после запятой) |
| `event_type` | `LowCardinality(String)` | Строка с оптимизацией для столбцов с небольшим числом уникальных значений — создаётся словарь, хранятся числовые индексы |
| `user_id` | `UInt64` | Беззнаковое 64-битное целое число, подходит для идентификаторов |
| `ip_address` | `IPv4` | Специальный тип для IPv4-адресов, хранится как 32-битное целое |
| `url` | `String` | Строка произвольной длины |
| `duration_ms` | `UInt32` | Беззнаковое 32-битное целое для продолжительности в миллисекундах |

**Параметры движка:**

- `ENGINE = MergeTree()` — основной движок для аналитических таблиц с сортировкой по первичному ключу.
- `ORDER BY (timestamp, user_id)` — ключ сортировки; данные на диске отсортированы сначала по `timestamp`, затем по `user_id`. Запросы с `WHERE` по этим столбцам будут выполняться значительно быстрее.

### Вставка данных

```sql
INSERT INTO access_logs VALUES
    ('2024-06-19 10:00:00.123', 'page_view', 101, '192.168.1.1', '/home', 150),
    ('2024-06-19 10:00:01.456', 'click',     101, '192.168.1.1', '/button_a', 20),
    ('2024-06-19 10:00:02.789', 'page_view', 102, '10.0.0.5',   '/products', 300);
```

> **Важно.** `INSERT` в ClickHouse оптимизирован для массовой вставки. Вставка по одной строке неэффективна. В production данные вставляются батчами или потоками из Kafka, S3 и т.д.

### Аналитический запрос

```sql
SELECT
    event_type,
    count(DISTINCT user_id) AS unique_users,
    avg(duration_ms)        AS avg_duration_ms
FROM access_logs
GROUP BY event_type;
```

## Типы данных ClickHouse — обзор

### Числовые типы

| Категория | Типы | Рекомендация |
|-----------|------|-------------|
| Целые числа | `Int8`/`UInt8` … `Int256`/`UInt256` | Выбирайте наименьший тип, вмещающий ваши значения |
| С плавающей точкой | `Float32`, `Float64` | Для вычислений, где допустима потеря точности |
| Десятичные | `Decimal32`, `Decimal64`, `Decimal128` | Для денежных сумм и точных вычислений |

### Строковые типы

| Тип | Назначение |
|-----|-----------|
| `String` | Строка произвольной длины |
| `FixedString(N)` | Строка фиксированной длины (хэши, коды стран) |
| `LowCardinality(String)` | Столбцы с малым числом уникальных значений — словарное кодирование |

### Дата и время

| Тип | Точность |
|-----|---------|
| `Date` | Только дата |
| `DateTime` | Дата и время (до секунды) |
| `DateTime64(precision)` | Дата и время с точностью до милли-/микро-/наносекунд |

> **Рекомендация:** всегда храните временные метки в UTC.

### Прочие типы

| Тип | Назначение |
|-----|-----------|
| `Boolean` (`UInt8`) | Логическое значение (0 или 1) |
| `UUID` | Универсальный уникальный идентификатор |
| `IPv4`, `IPv6` | Специализированные типы для IP-адресов |
| `Array(T)` | Массив значений одного типа |
| `Nullable(T)` | Позволяет хранить NULL; используйте только при необходимости |

## Движки таблиц семейства MergeTree

### Базовый MergeTree

```sql
CREATE TABLE my_table (
    col1 DataType,
    col2 DataType
)
ENGINE = MergeTree()
PARTITION BY expression     -- (опционально) выражение для партиционирования
ORDER BY (col1, col2)       -- ключ сортировки (ОБЯЗАТЕЛЕН)
PRIMARY KEY (col1)          -- (опционально) подмножество ORDER BY для первичного индекса
TTL expression              -- (опционально) время жизни данных
SETTINGS setting = value;   -- (опционально) дополнительные настройки
```

**Ключевые параметры:**

- `PARTITION BY` — разбиение на партиции (чаще всего по дате: `toYYYYMM(timestamp)`).
- `ORDER BY` — сортировка данных внутри партиции; основной индекс.
- `PRIMARY KEY` — в ClickHouse **не гарантирует уникальность**; определяет столбцы разреженного индекса.

### Вариации MergeTree

| Движок | Назначение | Пример использования |
|--------|-----------|---------------------|
| `ReplacingMergeTree([ver])` | Дедупликация: при слиянии оставляет последнюю версию строки по ORDER BY | Таблицы справочников, профили пользователей |
| `SummingMergeTree([cols])` | Суммирование числовых столбцов при слиянии строк с одинаковым ORDER BY | Агрегация метрик: просмотры, клики, суммы |
| `AggregatingMergeTree` | Хранение промежуточных состояний агрегатных функций | Предварительная агрегация `uniqCombined`, `quantiles` |
| `CollapsingMergeTree(sign)` | Удаление «парных» строк по столбцу Sign | Отслеживание сессий, состояний |
| `Replicated*MergeTree` | Репликация данных между серверами кластера (ZooKeeper / ClickHouse Keeper) | Production-системы с отказоустойчивостью |

### Другие категории движков

| Категория | Примеры | Когда использовать |
|-----------|---------|-------------------|
| Лог-движки | `Log`, `TinyLog`, `StripeLog` | Временные таблицы, небольшие данные, отладка |
| Внешние системы | `Kafka`, `MySQL`, `PostgreSQL`, `S3`, `URL`, `File` | Чтение из внешних источников без импорта |
| Специальные | `Dictionary`, `Distributed`, `Buffer` | Словари в RAM, распределённые запросы, буферизация |

## Практика. MergeTree с партиционированием

### Создание таблицы с партиционированием

```sql
DROP TABLE IF EXISTS my_first_db.access_logs;

CREATE TABLE my_first_db.access_logs (
    timestamp    DateTime64(3),
    event_type   LowCardinality(String),
    user_id      UInt64,
    ip_address   IPv4,
    url          String,
    duration_ms  UInt32
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(timestamp)
ORDER BY (timestamp, user_id);
```

Здесь `toYYYYMM(timestamp)` извлекает год и месяц (например, `202406` для июня 2024). ClickHouse создаст отдельные директории для каждой партиции.

### Вставка данных за разные месяцы

```sql
INSERT INTO my_first_db.access_logs (timestamp, event_type, user_id, ip_address, url, duration_ms) VALUES
    ('2024-05-15 10:00:00.123', 'page_view', 100, '192.168.1.1',  '/old_home',        150),
    ('2024-05-15 10:00:01.456', 'click',     100, '192.168.1.1',  '/old_button_a',     20),
    ('2024-06-19 10:00:00.123', 'page_view', 101, '192.168.1.1',  '/home',            150),
    ('2024-06-19 10:00:01.456', 'click',     101, '192.168.1.1',  '/button_a',         20),
    ('2024-06-19 10:00:02.789', 'page_view', 102, '10.0.0.5',     '/products',        300),
    ('2024-06-20 10:00:03.000', 'page_view', 101, '192.168.1.1',  '/contact',         100),
    ('2024-06-20 10:00:04.111', 'click',     102, '10.0.0.5',     '/product_details',  50),
    ('2024-07-01 08:00:05.222', 'page_view', 103, '172.16.0.10',  '/home',            200),
    ('2024-07-01 08:01:05.222', 'page_view', 104, '172.16.0.11',  '/home',            250);
```

### Проверка партиций

```sql
SELECT
    partition,
    name,
    rows,
    bytes_on_disk
FROM system.parts
WHERE table = 'access_logs' AND database = 'my_first_db' AND active;
```

Вы увидите партиции `202405`, `202406` и `202407`.

### Запрос с фильтрацией по партиции

```sql
SELECT
    event_type,
    count()          AS events,
    avg(duration_ms) AS avg_duration
FROM my_first_db.access_logs
WHERE timestamp >= '2024-06-01' AND timestamp < '2024-07-01'
GROUP BY event_type
ORDER BY events DESC;
```

ClickHouse автоматически определит, что нужно читать только партицию `202406`.

## Практика: ReplacingMergeTree

### Создание таблицы профилей пользователей

```sql
CREATE DATABASE IF NOT EXISTS my_app_data;

CREATE TABLE my_app_data.user_profiles (
    user_id              UInt64,
    username             String,
    email                String,
    registration_date    Date,
    last_login_timestamp DateTime,
    is_premium           Nullable(UInt8),
    region               LowCardinality(String),
    version              UInt64
)
ENGINE = ReplacingMergeTree(version)
ORDER BY (user_id, registration_date);
```

### Вставка данных с дубликатами

```sql
INSERT INTO my_app_data.user_profiles VALUES
    (1, 'alice',   'alice@example.com',   '2023-01-15', '2024-06-01 10:00:00', 1,    'Europe',  1),
    (2, 'bob',     'bob@example.com',     '2023-03-20', '2024-06-15 14:30:00', 0,    'Asia',    1),
    (3, 'charlie', 'charlie@example.com', '2023-06-10', '2024-05-20 09:00:00', NULL, 'America', 1);

-- Обновление записи для alice (version = 2)
INSERT INTO my_app_data.user_profiles VALUES
    (1, 'alice', 'alice_new@example.com', '2023-01-15', '2024-07-10 18:00:00', 1, 'Europe', 2);
```

### Проверка дедупликации

```sql
-- До слияния — могут быть видны обе версии
SELECT * FROM my_app_data.user_profiles WHERE user_id = 1;

-- Принудительное слияние (НЕ используйте в production без понимания последствий!)
OPTIMIZE TABLE my_app_data.user_profiles FINAL;

-- После слияния — останется только версия 2
SELECT * FROM my_app_data.user_profiles WHERE user_id = 1;
```

## Индивидуальные варианты заданий

**Номер варианта** определяется по формуле:  
`Вариант = (порядковый_номер_в_журнале - 1) % 200 + 1`

Все задания выполняются на облачном сервере `envlab.ru:9191` с выданными лектором учётными данными.

> **Правило именования объектов:**  
> Все базы данных и таблицы должны содержать номер варианта в имени.  
> Например, для варианта 42: `db_var042`, `sales_var042`, `products_var042` и т.д.

### Задание 1. Создание базы данных и таблицы продаж

Создайте базу данных `db_varNNN` (где `NNN` — ваш трёхзначный номер варианта с ведущими нулями).

В ней создайте таблицу `sales_varNNN` со следующей структурой:

```sql
CREATE TABLE db_varNNN.sales_varNNN (
    sale_id        UInt64,
    sale_timestamp DateTime64(3),
    product_id     UInt32,
    category       LowCardinality(String),
    customer_id    UInt64,
    region         LowCardinality(String),
    quantity       UInt16,
    unit_price     Decimal64(2),
    discount_pct   Float32,
    is_online      UInt8,
    ip_address     IPv4
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(sale_timestamp)
ORDER BY (sale_timestamp, customer_id, product_id);
```

Вставьте **не менее 100 строк** данных, распределённых по **не менее чем 3 разным месяцам**.

**Требования к данным по вариантам:**

- `sale_id` начинается с `NNN * 1000 + 1` (например, для варианта 42: с `42001`).
- `customer_id` лежит в диапазоне `[NNN * 100, NNN * 100 + 99]` (для варианта 42: от `4200` до `4299`).
- `product_id` лежит в диапазоне `[NNN * 10, NNN * 10 + 19]` (для варианта 42: от `420` до `439`).
- `unit_price` — от `(NNN % 50 + 10).00` до `(NNN % 50 + 510).00`.
- `quantity` — от `1` до `(NNN % 20 + 5)`.

### Задание 5. Комплексный анализ и самопроверка

Выполните следующие запросы и зафиксируйте результаты:

1. **Проверка партиций** таблицы `sales_varNNN`:
   ```sql
   SELECT
       partition,
       count()        AS parts,
       sum(rows)      AS total_rows,
       formatReadableSize(sum(bytes_on_disk)) AS size
   FROM system.parts
   WHERE database = 'db_varNNN'
     AND table = 'sales_varNNN'
   ```

   ```sql
       p.product_name,
       p.category,
       sum(s.quantity * s.unit_price * (1 - s.discount_pct)) AS revenue
   FROM db_varNNN.sales_varNNN AS s
   INNER JOIN db_varNNN.products_varNNN AS p
       ON s.product_id = p.product_id
   GROUP BY p.product_name, p.category
   ORDER BY revenue DESC
   LIMIT 5;
   ```

3. **Типы данных** — выведите структуру всех трёх созданных таблиц:

4. **Запрос с массивом** — создайте временную таблицу `tags_varNNN` с колонкой `Array(String)` и выполните запрос с `arrayJoin`:

   ```sql
   CREATE TABLE db_varNNN.tags_varNNN (
       item_id  UInt32,
       item_name String,
       tags     Array(String)

   INSERT INTO db_varNNN.tags_varNNN VALUES
       (1, 'Item A', ['sale', 'popular', 'new']),
       (2, 'Item B', ['premium', 'limited']),
       (3, 'Item C', ['sale', 'clearance']);

   SELECT
       arrayJoin(tags) AS tag,
       count()         AS items_count
   FROM db_varNNN.tags_varNNN
   GROUP BY tag
   ORDER BY items_count DESC;
   ```

5. **Контрольная сумма** — итоговая проверка:

   ```sql
   UNION ALL
   SELECT
       'products', count(), sum(toUInt64(product_id)) FROM db_varNNN.products_varNNN FINAL
   ```
